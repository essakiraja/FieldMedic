# FieldMedic

Emergency medical guidance powered by AI — voice, vision, and offline-first.

FieldMedic turns any smartphone into an emergency medical assistant. A bystander points their camera at a wound, speaks or types what they see, and receives calm step-by-step guidance in their own language. A remote doctor monitors the case in real time and can send instructions directly to the bystander's device, where they are spoken aloud.

The system works on a $30 Android phone over 2G. It works offline. It works in 20 languages. That is the point.

---

## The Problem

Every year, hundreds of thousands of people die from injuries that were survivable. Not because help was unavailable — but because the person standing next to them did not know what to do. Bystanders hesitate, act incorrectly, or do nothing. The gap between injury and professional help is often the difference between life and death.

FieldMedic closes that gap.

FieldMedic is a real-time multimodal agent that handles interruptions, responds in the user's language, adapts its guidance based on changing conditions, and coordinates between two people (bystander and doctor) across two devices simultaneously.

## How It Works

A bystander opens FieldMedic on any phone. They can:

- Point the camera at a wound — Gemini Vision analyzes it and returns a severity assessment, injury type, and the single most important first action
- Speak what they see in any language — the intake agent detects the language and routes the request accordingly
- Follow step-by-step voice instructions generated by the guidance agent in their own language
- Receive adaptive check-in questions that adjust the guidance if the situation changes
- Connect with a remote doctor who can type instructions that are spoken aloud on the bystander's device

A doctor opens the responder dashboard on any device with a browser. They see all active cases, triage assessments, wound photos, GPS location, and the full conversation transcript. They can send instructions directly into the bystander's guidance flow.

```
Bystander (phone)
    |
    | photo + voice input
    v

    v                              v
```

## Agent Architecture (ADK)

Three specialized agents coordinated by a single orchestrator:

**IntakeAgent** receives raw user text in any language and returns a structured context object: detected language, emotional state, urgency level, and injury category. This runs before any other agent so that all downstream responses are in the correct language.

**TriageAgent** receives a base64-encoded wound photo and the enriched context from the intake agent. It calls Gemini 2.5 Flash with the image and a clinical prompt, returning severity, injury type, immediate dangers, the most critical first action, and approximate wound coordinates for the heatmap overlay. If vision analysis fails, it returns a conservative fallback rather than nothing.

**GuidanceAgent** receives the triage result and generates 4-8 clear numbered instructions in the detected language. It enforces the language in the system prompt explicitly. If the API is unreachable, it falls back to a pre-loaded offline decision tree covering six trauma categories.

The orchestrator runs intake then triage sequentially (triage depends on intake output), then guidance. A combined pipeline method runs all three in a single call for the most common flow.

## Google Cloud Services

**Cloud Run** hosts the FastAPI backend. Auto-scales to zero when idle, scales on demand during active cases. Deployed via Cloud Build CI/CD on every push to main.

**Firestore** stores case records, chat messages between bystander and doctor, and guidance steps delivered. Used as the real-time channel for the two-device chat feature — doctor messages written to Firestore are polled by the bystander's device and spoken aloud.

**Cloud Storage** stores wound photos captured during triage. Base64 images are uploaded to Cloud Storage before being written to Firestore, keeping the database clean.

**Cloud TTS** synthesizes guidance step audio using Neural2 voices. Supports language-specific voices for English, Swahili, German, French, Arabic, Hindi, Spanish, and Portuguese. Results are cached in memory to avoid re-synthesizing identical text.

**Cloud Build** runs the deployment pipeline. The `cloudbuild.yaml` in the repository builds the Docker image, pushes it to Artifact Registry, and deploys to Cloud Run on every commit.

**Firebase Hosting** serves the React PWA frontend from a global CDN at `https://medic-4053.web.app`.

## Offline Mode

The Service Worker pre-caches all static assets and the offline decision tree on first load. When the backend is unreachable:

The `useOfflineMode` hook detects network loss via browser events and active health probes against the `/api/health` endpoint. On failure, the connection quality switches to `offline` and the app activates the local decision tree immediately — no error, no blank screen.

The offline tree covers six trauma categories with clinically reviewed step sequences: bleeding, burns, choking, fracture, cardiac arrest, and shock. Each step is spoken via the browser Web Speech API, which requires no network. Wound photos captured offline are stored locally and synced to Cloud Storage when connectivity returns.

---

## Multilingual Support

Language detection runs in the intake agent on every request. The detected language code propagates through the entire pipeline — the guidance agent is explicitly instructed to respond in that language, Cloud TTS selects the appropriate Neural2 voice, and the browser speech engine uses a matched locale.

The system does not use a translation layer. Gemini 2.5 Flash handles multilingual input and output natively, which means idiomatic phrasing, appropriate medical terminology, and natural sentence structure in each language rather than translated English.

---

## Adaptive Guidance

After key steps in the guidance flow, the agent asks a check-in question — for example, after the third bleeding control step: "Is the bleeding slowing down?" The bystander answers yes or no. The guidance agent fetches updated steps with the check-in answer embedded in the triage context, so subsequent instructions adapt to the actual situation rather than following a fixed script. A `firedRef` set prevents any trigger from firing more than once per session.

---

## Two-Device Chat

A doctor opens the responder dashboard and selects an active case. A `ChatPanel` component is embedded in the detail view. Messages sent from the doctor are written to `cases/{id}/messages` in Firestore and polled every three seconds by the bystander's device. New doctor messages are automatically spoken aloud via the same TTS pipeline used for guidance steps — the bystander hears the doctor's instructions without needing to read anything. Bystander replies appear in the doctor's chat panel in real time.

## Setup and Running

### Prerequisites

- Node.js 20+
- Python 3.12+
- Google Cloud project with billing enabled
- gcloud CLI authenticated

### Required APIs

Enable these in your GCP project:

```bash
gcloud services enable \
  run.googleapis.com \
  cloudbuild.googleapis.com \
  firestore.googleapis.com \
  storage.googleapis.com \
  texttospeech.googleapis.com \
  secretmanager.googleapis.com
```

### Environment variables

Copy `.env.example` to `.env` and fill in:

```
VITE_API_BASE_URL=https://your-cloud-run-url.run.app
VITE_APP_URL=https://your-project.web.app
VITE_FIREBASE_API_KEY=...
VITE_FIREBASE_AUTH_DOMAIN=your-project.firebaseapp.com
VITE_FIREBASE_PROJECT_ID=your-project-id
```

Backend (set as Cloud Run environment variables or in `backend/.env` for local):

```
GEMINI_API_KEY=...
GCP_PROJECT_ID=your-project-id
GCP_REGION=us-central1
FIRESTORE_DB=(default)
STORAGE_BUCKET=your-bucket-name
```

### Local development

```bash
# Backend
cd backend
pip install -r requirements.txt
uvicorn main:app --reload --host 0.0.0.0 --port 8000

# Frontend (separate terminal)
npm install
npm run dev
```

### Deploy backend to Cloud Run

```bash
gcloud run deploy fieldmedic-api \
  --source ./backend \
  --region us-central1 \
  --allow-unauthenticated \
  --set-env-vars "GCP_PROJECT_ID=your-project-id,GEMINI_API_KEY=your-key"
```

Or use the automated pipeline:

```bash
gcloud builds submit --config cloudbuild.yaml
```

### Deploy frontend to Firebase Hosting

```bash
npm run build
firebase deploy --only hosting
```

### Authorized responder emails

Edit `src/config/firebase.js` and add email addresses to `ALLOWED_EMAILS`. Only these accounts can access the responder dashboard and case history pages.
